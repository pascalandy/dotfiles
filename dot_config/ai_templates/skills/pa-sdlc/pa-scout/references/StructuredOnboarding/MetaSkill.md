# StructuredOnboarding

## When To Use

Use this mode when the user wants more than a scan. The goal here is to help a newcomer become effective quickly.

This mode is for human learning and handoff, not just situational awareness.

Typical triggers:

- Onboard me to this
- Walk me through this system
- Prepare a newcomer guide
- Explain how someone should navigate this workspace

## Core Method

1. Start from a broad orientation.
2. Read the minimum set of files or artifacts needed to explain the system clearly.
3. Explain the system from the outside in.
4. Map organization, concepts, and common tasks.
5. Create a durable guide only when the user asks for one.

## Subject Adaptation Rules

- For a product or website, describe the user or editor experience before internal structure.
- For a developer-facing system, explain how contributors or consumers interact with it.
- For PM or operations workspaces, explain how work enters, moves, and completes.
- For personal knowledge systems, explain how information is organized, discovered, and maintained.

## Workflow

1. Reuse the high-signal findings from broad orientation.
2. Identify the audience for the onboarding guide.
3. Explain purpose, structure, important concepts, and common tasks.
4. Show where to look for specific changes or questions.
5. If requested, write a durable onboarding artifact.

## Output Format

When answering inline, use these sections:

1. `What This Is`
2. `How Someone Uses It`
3. `How It Is Organized`
4. `Key Concepts`
5. `Common Tasks`
6. `Where To Look`
7. `Evidence`
8. `Unknowns`

When writing a durable artifact, keep the same structure but expand examples and references.

## Examples

- "Onboard a new engineer to this repo."
- "Prepare a guide for someone inheriting this content workspace."

## Boundaries

- Optimize for human clarity over exhaustive coverage.
- Do not overwrite existing onboarding artifacts unless the user asks.
- Do not collapse into a quick scout brief when the request clearly needs a teachable walkthrough.
- Do not turn onboarding into architecture redesign or critique.
