# TargetedInquiry

## When To Use

Use this mode when the user already has a specific question and does not need a broad tour.

Typical triggers:

- Where is this handled?
- How does this workflow work here today?
- Research this area
- Investigate this and explain what currently exists

## Core Method

1. Clarify the research question.
2. Decompose it into the smallest useful search areas.
3. Inspect only the relevant artifacts.
4. Synthesize what currently exists with direct evidence.
5. State unknowns or misframed assumptions explicitly.

## Subject Adaptation Rules

- For code, track the primary files, modules, entry points, and interfaces that answer the question.
- For content systems, track the relevant pages, templates, taxonomies, approvals, and publishing flows.
- For PM systems, track the relevant artifacts, states, conventions, and transitions that answer the question.
- For knowledge systems, track note clusters, indexes, links, naming patterns, and missing bridges.

## Workflow

1. Restate the scoped research question.
2. Identify candidate evidence sources.
3. Investigate in parallel when the sub-questions are independent.
4. Synthesize the answer with file, artifact, or location references.
5. Recommend the next scout move only if it helps answer the question.

## Output Format

Produce these sections:

1. `Research Question`
2. `Summary`
3. `Detailed Findings`
4. `Evidence`
5. `Unknowns Or Misframed Assumptions`
6. `Recommended Next Scout Step`

## Examples

- "How are approvals handled in this workspace today?"
- "Where is the publishing flow defined in this system?"

## Boundaries

- Stay descriptive first. Explain what is there before suggesting changes.
- Do not broaden into a full onboarding pass unless the evidence forces it.
- Do not turn this into a change map, blast-radius analysis, or implementation plan.
- Do not fabricate rationale that the current-state evidence cannot support.
