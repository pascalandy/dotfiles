# Vision — Ontology Layer for wiki-map

Mode: `AlignmentDraft`. Direction is settled; this artifact makes the target state explicit before architecting the workflow.

## Request

Extend `dot_config/ai_templates/skills/knowledge/wiki-map` with an auto-generated ontology layer that surfaces the conceptual landscape of large wikis. The AI assistant maintains it transparently; the user never hand-edits these files.

## Current State

`wiki-map` exposes wiki content through:

- `INDEX.md` — structural catalog, per-page, mechanical
- `[[wikilinks]]` + computed backlinks — raw graph edges
- `sources:` and `contradictions:` frontmatter — provenance
- `references/_meta/topic-map.md` — reserved for wikis over 200 entries

What is missing once a wiki passes ~50 pages:

- No view of *what concepts* the wiki actually contains
- No view of *which pages cluster into systems*
- No view of *what questions the wiki implies but does not answer*
- No view of *the nature* of relations between pages, only that they exist

The structural layer is healthy. The semantic layer is invisible.

## Desired End State

A wiki of 50+ pages contains a `_meta/ontology/` folder, regenerated by the AI on demand, that gives the next session a fast semantic read of the wiki without reading every page.

```text
{wiki-name}/
  INDEX.md                       # existing structural catalog
  references/
    LOG.md
    {pages}.md
    _meta/
      topic-map.md               # existing
      ontology/                  # new
        INDEX.md                 # routes to the four below
        concepts.md              # core ideas + 1-line definitions
        systems.md               # named clusters of pages forming a system
        questions.md             # gaps the wiki implies but does not answer
        connections.md           # relation types between pages (optional)
```

The ontology folder is a derived artifact, like `_backlinks.json` in similar systems. Lint flags hand-edits as drift.

## Key Decisions

| Decision | Choice | Reasoning |
|---|---|---|
| File count | 4, not 6 | `full-wiki-ontology.md` and `sources-ontology.md` collapse into INDEX + existing `sources:` frontmatter |
| Maintenance model | Auto-regenerated | Hand-maintained ontology rots faster than the wiki itself |
| Threshold | Wikis with 50+ pages | Below 50, INDEX + wikilinks are sufficient; ontology adds noise |
| Trigger | Explicit `wiki-map ontology` command, or final step of `FullSweep --deep` | Refreshing on every ingest is too expensive |
| Location | `_meta/ontology/` | Reuses the existing `_meta/` namespace for derived views |
| Format | Markdown tables + wikilinks | Legibility for the next AI session beats formal graph formats (RDF, OWL, JSON-LD) |
| New tag | `kind/ontology` | Lint exempts these files from orphan and outbound-link checks, like `kind/log` and `kind/wiki` |
| Workflow placement | New workflow under `Lint/workflows/Ontology.md` | Fits the existing sub-skill structure |
| Architecture share with Breakdown | Shared mining phase | Same parallel-subagent extraction feeds both: Ontology produces read-only views, Breakdown produces real pages |

## Success Signals

- A new AI session opens the wiki, reads `_meta/ontology/INDEX.md` plus one of the four files, and can navigate semantically without reading every page.
- `questions.md` surfaces real gaps that lead to follow-up ingest sessions.
- `systems.md` exposes clusters the user did not consciously design.
- Lint detects hand-edits to ontology files as drift.
- The ontology workflow does not run on small wikis and does not slow down ingest.

## Risks and Review Points

| Risk | Mitigation |
|---|---|
| Ontology rots if auto-refresh is forgotten | Trigger as a final step of `FullSweep --deep`; include freshness check in `QuickCheck` |
| 4 files become noise on medium-size wikis | Hard threshold at 50 pages; below it, the workflow exits without writing |
| Mining cost is high on every refresh | Parallel subagent batches; cache per-page extraction by `date_updated` |
| `connections.md` duplicates wikilinks without adding value | Only generate if extraction captures relation *type*, not just edge existence; otherwise drop the file |
| Schema sprawl from `kind/ontology` | Single new tag, exempt from existing checks; documented once in `SCHEMA.md` |

## Recommended Next Phase

`pa-architect` on the Ontology workflow — design the mining phase shared with Breakdown, the per-file generation logic, the freshness model, and the schema additions to `SCHEMA.md`.
